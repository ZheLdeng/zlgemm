# current_m8_lite_80c_opt_parts_v2 分析报告

数据文件：`results/m8/current_m8_lite_80c_opt_parts_v2.xlsx`

本报告只基于 v2 工作簿中的 `parts`、`tail`、`dispatch` 三个 sheet。v2 已经包含 BF16/I8、SVE/NEON 四类实现；`dispatch` 包含 `auto/m/n/2d/nblock/ngroup` 等拆分策略；`parts` 已补齐 I8 的 `full/prepacked/noload/nostore/compute_only` 拆分归因。

基准单核性能使用 `cpufb/机器硬件性能.md` 中的硬件测试值：

| 类型 | 单核 baseline |
|---|---:|
| SVE BF16 | 92.72 GFLOPS |
| NEON BF16 | 92.704 GFLOPS |
| SVE I8 | 370.82 GOPS |

v2 的多线程小规模剪枝已经生效：当 `ABC bytes / threads < 256KB` 时不再跑多线程组合。因此下面的“每线程块大小”统计不会覆盖小于 L1 的多线程极小块。

关键发现：
- `2048x4096x64`（M 大 N 极小）各线程下 full/compute 稳定在 ~86%，kernel 效率高，dispatcher 主要收益来自选对切分方向。

| shape | threads | best split | best / auto |
|---|---:|---|---:|
| 64x4096x64 | 4 | n | 1.45x |
| 512x4096x1024 | 40 | 2d | 1.24x |
| 512x4096x1024 | 20 | 2d | 1.24x |
| 2048x512x4096 | 64 | 2d | 1.22x |
| 128x4096x1024 | 20 | 2d | 1.22x |

已同步到代码的策略：`BF16_SVE_SPLIT` 未显式指定时，`threads>=10`、`K>=512` 且 M/N work unit 足够时自动启用 2D split；环境变量仍可强制 `m/n/2d/ngroup/nblock` 做对照。

### 1.4 每线程规模关系

v2 数据里，多线程小块已剪枝，所以规律主要来自 256KB/thread 以上的块。SVE BF16 对每线程块大小的依赖仍然明显：`>L2` 或 `>3/4L2` 的大块可以维持 90% 左右；中等块在 80 核时更容易掉到 70% 左右。

| 分桶 | 典型结论 |
|---|---|
| M 小 | 当 N/K 足够大时仍能跑高；如果同时 N 小，容易出现 task 粒度不足 |
| M 中 | 最稳定，8-40 核通常接近 85%-90%/core |
| M 大 | 大线程数下受 store 和内存层级压力影响更强 |
| K 小 | 更依赖 M/N 是否足够大，否则调度开销占比上升 |
| K 大 | compute-only 更稳，full 更容易暴露 AB load 和 C store |
| N 小 | 对切 N 不友好，适合切 M 或 2D |
| N 大 | SVE BF16 最容易保持高利用率 |

- `512x4096x1024` 的 8T 天花板只有 54.0%（SVE 是 71.1%），解释了为什么 m split 在这个 shape 上能带来 1.7-2.7x 的提升：换 m split 减少 A 重复 pack 后，kernel 能够更接近其天花板。

已同步到代码的策略：高线程、大 M/K，且 N 没有宽到足以抵消重复 A 数据准备时，NEON BF16 auto 更偏向 M split。当前规则覆盖 `M>=512,K>=512,N<=4M`，并额外覆盖 `M>=128,K>=2048,N<=1024` 的中 M 大 K 场景。

### 2.4 同精度 SVE 对比 NEON

BF16 下，低线程 SVE 和 NEON 接近；32 核以上 SVE full/dispatcher 更有优势。按 best split 看，80 核 SVE BF16 约为 NEON BF16 的 1.2x；按 auto 看，80 核约为 1.4x。

- 所有 shape×线程组合的 full/compute 都在 42%-54% 窄区间内，远低于 BF16 的波动范围。这印证了 dispatcher split 调整收益极小的结论：无论怎么切分，kernel 自身的 load/store 瓶颈都在同一水平。
- `2048x1024x8192` 在 80T 时仍有 49.9%（BF16 仅 21-41%），因为 I8 的 compute-only 绝对值更高（约 4x），full 更快触达 load/store 墙，反而让天花板更"平"。
- auto median 的 38.2%-51.2% vs baseline 与 full/compute 天花板接近，说明 dispatcher 已经基本榨干了当前 kernel 能提供的空间。后续优化应集中在 kernel 的 AB load 排布和预取。

最优 split 分布：`n:29`、`auto:27`、`m:5`。最大收益也只有约 1.07x，说明调度方向不是 SVE I8 当前最大问题，kernel 的 load/store 数据通路更值得优先优化。

代码策略：SVE I8 的 auto 暂不调整；继续保留 `m/n` 强制测试入口用于验证特定机器。

### 3.4 每线程规模关系

SVE I8 full 的利用率整体稳定在 45%-52% 区间，和每线程块大小的相关性弱于 BF16。原因是 compute-only 峰值更高，full 更快进入 load/store 受限状态；即使每线程块较大，也不一定显著提升 full/compute。

## 4. NEON I8

### 4.1 compute-only 线性度

NEON I8 的 auto 在部分大 shape 上不是最优，`m` split 对若干大 M/K/N 场景有非常明显收益。

关键发现：
- `2048x4096x64` 在所有线程下天花板都 >84%，dispatcher 在这个 shape 上几乎不需要关心 split 策略，kernel 自身效率足够高。
- `2048x1024x8192` 在 80T 时天花板仅 15.4%，是全部四类实现中最低的。auto 的 31.6% vs baseline 在此 shape 上实际远超 kernel 天花板——这说明测量中 auto 在此 shape 下的中位值被其他更高效率的 shape 拉高了，该 shape 自身的 dispatcher 表现远低于 auto median。

- 对比 `2048x1024x8192` 的 prepacked 仍有 89.5%（见 4.2 节最敏感场景表），说明该 shape 天花板 15.4% 的核心原因是在线 reorder/AB load——这部分通过 split 无法解决，只能靠预打包或减少在线 pack 开销。

| shape | threads | best split | best / auto |
|---|---:|---|---:|
| 512x4096x1024 | 32 | m | 2.73x |
| 2048x512x4096 | 80 | m | 2.50x |
| 2048x512x4096 | 64 | m | 2.49x |
| 512x4096x1024 | 40 | m | 2.03x |
| 512x4096x1024 | 16 | m | 1.72x |

已同步到代码的策略：高线程、大 M/K，且 `N<=2M` 时 NEON I8 auto 改为优先 M split；`N>=4M` 的极宽 N 仍保持 N/auto，因为 v2 数据里 `2048x1024x8192` 这类宽 N 下 N split 仍有竞争力。

### 4.4 同精度 SVE 对比 NEON

I8 下，低线程 NEON full 明显强于 SVE full；高线程 auto 下 SVE 看起来更强，但这主要是 NEON auto 在大 shape 上选 split 不理想。按 best split 看，80 核 SVE I8 和 NEON I8 接近，NEON 甚至略高。因此 I8 的首要优化不是简单切换 SVE/NEON，而是让 dispatcher 对 NEON I8 的大 shape 更偏向 `m` split，并降低在线 reorder/store 开销。

## 5. tail 测试

tail sheet 覆盖 `tail_M=17/18/20`，对应 M8 主体后走 `+1/+2/+4` tail。整体结论：

| 实现 | 结论 |
|---|---|
| NEON | 多数线程下 tail 约为 base 的 99%-100%；高线程局部有 90% 或超过 100% 的波动，更多来自 split/task balance 和测量噪声 |
| SVE | 低到中线程下 `+1/+2/+4` 通常在 95%-100%；32/40 核的 `+1/+2` 掉到约 75%，说明尾块在多线程切分下容易造成任务粒度和负载均衡问题 |

tail 对总性能的影响不能只看 tail kernel 本身；当 M 很小或者线程数高时，tail 会改变每线程分到的 M8 block 数量，进而影响调度和同步开销。

## 6. 优化方向

### 6.1 SVE BF16

优先优化 full path 的在线 A 数据准备、AB load 和 C store 的叠加压力。80 核大 shape 下 full 只有 compute-only 的 41%-67%，但 prepacked 能恢复到 93% 以上，说明预打包或更低成本 A reorder 对高线程 BF16 有价值。

建议：

1. 对 10 核以上且 M/N work unit 足够的 shape 自动尝试 `2d`。
2. 继续保留 `prepacked/noload/nostore/nozero` 作为归因项。
3. 针对 `2048x1024x8192` 和 `2048x512x4096` 做在线 A 数据准备、AB load、C store 的组合专项实验。

### 6.2 NEON BF16

NEON BF16 high-thread full 掉得明显，且 `m` split 对部分场景提升很大。auto 已加入大 M/K 场景偏向 `m` 的规则；后续重点复查 C store、bias/bf16 输出路径，以及是否可以像 I8 一样把 A pack 做成更适合多线程的离线/分块路径。

### 6.3 SVE I8

SVE I8 compute-only 已达硬件峰值，但 full 约 45%-51%。`prepacked≈full`，说明不是 reorder 单点问题；`noload` 高、`nostore` 接近 full，说明 full 更偏 AB load 受限。建议优先优化 B/A load 的调度、预取和 load/compute 交错，而不是继续调 split。

### 6.4 NEON I8

NEON I8 compute-only 很强，full 的主要问题是在线 reorder/pack 和 AB load。`prepacked` 能恢复到 80%-96%，说明预打包或减少在线 reorder 很有价值。同时 dispatcher 的 auto 对大 shape 不够好，已增加规则：

1. 大 M、大 K、N 不极宽且线程数高时优先尝试 `m` split。
2. 对 `512x4096x1024`、`2048x512x4096` 这类 shape 建立专门规则。
3. 保留 `prepacked` 作为关键 benchmark，因为它能清楚区分 reorder 与 kernel/store 开销。

## 7. 总结

1. BF16 compute-only SVE/NEON 都能接近单核 baseline；SVE BF16 full 在高线程下明显优于 NEON BF16。
2. SVE I8 compute-only 达到硬件峰值，但 full 仍明显受 load/store 限制。
3. NEON I8 compute-only 最稳，full 主要受在线 reorder/pack 和 AB load 限制；使用 best split 后高线程下可以接近甚至超过 SVE I8。
4. 当前 auto 已按 v2 数据调整：SVE BF16 高线程优先 2D，NEON BF16/NEON I8 的部分大 shape 优先 M split，SVE I8 暂不调整。
5. v2 的剪枝策略合理：低于 256KB/thread 的多线程 case 已经去掉，报告更能反映真实优化方向，而不是小任务调度噪声。
